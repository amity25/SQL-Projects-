);

 #For price DECIMAL(10,2) 

#10 (Precision): This is the total number of digits the column can store, including digits on both sides of the decimal point.
#2 (Scale): This is the number of digits specifically reserved for the fractional part (after the decimal point).


TABLE 3 — Orders
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    product_id INT,
    quantity INT,
    order_date DATE,

    FOREIGN KEY (customer_id)
    REFERENCES customers(customer_id),

    FOREIGN KEY (product_id)
    REFERENCES products(product_id)
);


Explanation for Primary Key and Foreign Key 

We add FOREIGN KEY and REFERENCES when one table is connected to another table.

This comes from understanding the relationship between tables in a database.

In your E-commerce project:

One customer can place many orders
One product can appear in many orders
One order can have one payment

So tables are not independent. They are connected.

Understanding the Logic

Think like this:

Customers Table
customer_id

This uniquely identifies every customer.

Example:

1 → Amit
2 → Priya
3 → Rahul

This column becomes the table's identity.

That is why we make it:

PRIMARY KEY
Now Look at Orders Table

Orders table needs to know:

Which customer placed this order?

So we add:

customer_id

inside orders table.

But this customer_id should only contain values that already exist in customers table.

That is where FOREIGN KEY comes in.

Meaning of FOREIGN KEY
FOREIGN KEY (customer_id)
REFERENCES customers(customer_id)

means:

The customer_id inside orders table must exist inside customers table.
Real-World Analogy

Imagine:

Customers Table = List of all students in school

Orders Table = Exam records

If exam records contain a student ID that does not exist in school records, data becomes invalid.

Foreign keys prevent this problem.

Why We Added These Foreign Keys
1. Customer Relationship
FOREIGN KEY (customer_id)
REFERENCES customers(customer_id)

Reason:
Every order must belong to a valid customer.

2. Product Relationship
FOREIGN KEY (product_id)
REFERENCES products(product_id)

Reason:
Every order must contain a valid product.

3. Payment Relationship
FOREIGN KEY (order_id)
REFERENCES orders(order_id)

Reason:
Every payment must belong to a valid order.

Simple Rule to Identify Foreign Keys

Ask this question:

"Is this column coming from another table?"

If YES → it is usually a foreign key.

STEP 6 — LEARN INNER JOIN

Basic syntax:

SELECT *
FROM table1
INNER JOIN table2
ON table1.column = table2.column;


PRACTICE JOIN


SELECT
    o.order_id,
    c.customer_name,
    p.product_name,
    o.quantity
FROM orders o
INNER JOIN customers c
ON o.customer_id = c.customer_id
INNER JOIN products p
ON o.product_id = p.product_id;

Explanation - Why Use o. ?

Because multiple tables are used in this query.

Sometimes different tables may have same column names like:

customer_id
product_id
order_id

So SQL needs clarity.

o. tells SQL exactly which table the column belongs to

c.customer_name,

Meaning

c is alias for customers table.

INNER JOIN customers c

So:
c.customer_name

means:
Take customer_name from customers table.

PROJECT TASK 1 — Revenue By Category

Business Question:
Which product category generates the highest revenue?

Query
SELECT
    p.category,
    SUM(o.quantity * p.price) AS total_revenue
FROM orders o
INNER JOIN products p
ON o.product_id = p.product_id
GROUP BY p.category
ORDER BY total_revenue DESC;

How the Query Works -

SUM(o.quantity * p.price) AS total_revenue: For every order, the database multiplies the number of items ordered (quantity) by the product's price (price). It then adds all of these subtotals together to find the total_revenue.
INNER JOIN ... ON ...: It links the orders table and products table together so that it can find the price of the specific product in each order.
GROUP BY p.category: It bundles the results into groups based on the category of the products (e.g., all "Electronics" orders are grouped into one sum, all "Clothing" orders into another).
ORDER BY total_revenue DESC: It sorts the final grouped results so the highest-grossing categories appear at the very top. 


What You Learn
SUM()
Revenue calculation
GROUP BY

PROJECT TASK 2 — Best-Selling Products

Business Question:
Which products sell the most?

Query
SELECT
    p.product_name,
    SUM(o.quantity) AS total_units_sold
FROM orders o
INNER JOIN products p
ON o.product_id = p.product_id
GROUP BY p.product_name
ORDER BY total_units_sold DESC;

How It Works - 

SELECT p.product_name: Displays the name of the product.
SUM(o.quantity): Adds up all instances of that product ordered in the orders table.
INNER JOIN... ON o.product_id = p.product_id: Matches the orders to their specific product details.
GROUP BY p.product_name: Groups the sales so the total quantity calculation applies to each specific product.
ORDER BY total_units_sold DESC: Sorts the final results from the highest number of units sold to the lowest.

Product analytics
Sales trends
Ranking products

PROJECT TASK 3 — Customer Lifetime Value

Business Question:
Which customers spend the most money?

Query
SELECT
    c.customer_name,
    SUM(pay.payment_amount) AS lifetime_value
FROM customers c
INNER JOIN orders o
ON c.customer_id = o.customer_id
INNER JOIN payments pay
ON o.order_id = pay.order_id
GROUP BY c.customer_name
ORDER BY lifetime_value DESC;

Key logic:

Joins: It links the customers table to their orders, then links those orders to the payments received.
Aggregation: SUM(pay.payment_amount) adds up every payment record associated with that specific customer name.
Grouping: GROUP BY c.customer_name ensures that all orders for "Alice Johnson" are collapsed into a single row.

PROJECT TASK 4 — Payment Trends

Business Question:
Which payment method is used most?

Query
SELECT
    payment_method,
    COUNT(*) AS total_transactions,
    SUM(payment_amount) AS total_amount
FROM payments
GROUP BY payment_method
ORDER BY total_amount DESC;

FROM customers c
INNER JOIN orders o
ON c.customer_id = o.customer_id
GROUP BY c.customer_name
HAVING total_orders > 1;
